# Fastest No-ID Cities — Menu Sections

**Date:** 2026-06-26
**Status:** Approved (design); pending implementation plan
**Component:** `vpn-dns-menubar` (Swift, StatusItemKit)

## Overview

Add two new sections to the VPN & DNS menu-bar app:

1. **Fastest US (No-ID)** — top 3 fastest US Mullvad cities located in states with
   no internet age-verification law for adult content.
2. **Fastest Non-US (No-ID · torrent-safe)** — top 3 fastest non-US Mullvad cities
   that are both age-verification-free and lenient on torrent/copyright enforcement.

Each section lists 3 cities ranked by measured latency. Clicking a city **toggles**
the VPN: connect to that city, or — if already connected to it — disconnect.

## Goals

- Surface the fastest privacy-appropriate exits as one-click (toggle) menu items.
- Keep the ranking honest: only ever rank on **direct** (tunnel-down) latency.
- Keep the candidate pool tied to the user's existing No-ID custom lists, and easy
  to refresh as those lists or Mullvad's fleet change.

## Non-Goals

- No runtime parsing of the No-ID custom lists (deferred; see Alternatives).
- No Mullvad split-tunnel integration for live-while-connected probing (deferred).
- No change to the existing dot/status/Tailscale behavior.

## Selection Criteria & Methodology

Two filters, both derived from the user's curation plus current legal reality:

### Filter 1 — No age-verification law (adult content)
Inherited from the user's **No-ID custom lists** (`No-ID L1/L2/L3`), which already
encode "places with no age/ID-verification laws." The candidate pool is exactly the
cities in those lists, split into US and non-US.

- US No-ID cities (from L1/L2): Boston MA, Chicago IL, Detroit MI, Secaucus NJ,
  Washington DC, Seattle WA.
- Non-US No-ID locations (from L1/L2/L3): Canada (all cities), Mexico, Colombia,
  Albania, Argentina, Chile, Israel, New Zealand, Peru, Philippines, Serbia,
  Thailand, Ukraine.

### Filter 2 — Lenient torrent/copyright enforcement
- **Within the US, copyright enforcement is federal and uniform** and does not vary
  by state; Mullvad forwards no DMCA notices regardless of exit. This filter
  therefore does **not** differentiate US cities — the US section ranks purely on
  Filter 1 + latency.
- For non-US countries it removes jurisdictions with an active enforcement regime.
  Of the No-ID pool, only **New Zealand** has one (Copyright Infringing File Sharing
  Act: three-notice Copyright Tribunal, fines to NZ$15k). **New Zealand is excluded.**
  (It is also ~205 ms away, so it would never reach top 3 regardless.) All other
  No-ID countries have weak/rare individual enforcement and stay in the pool.

### Latency
"Fastest" = lowest min ICMP RTT from the user's location, measured with the Mullvad
tunnel **disconnected** (see Latency Probe). One representative relay is pinged per
city to rank cities; the city item connects to the city (Mullvad picks the relay).

## Candidate Pool (P-A: baked allowlist + regen script)

A JSON resource bundled with the app lists each qualifying city once, with a
representative relay IP and a seed latency. Generated by
`scripts/refresh-candidates.sh`, which: reads the No-ID lists, applies the NZ/torrent
exclusion and US/non-US split, picks a representative relay per city, pings them
tunnel-down, and writes the JSON with fresh seeds. Refreshing the pool = re-run the
script + rebuild.

**Allowlist schema** (`Resources/candidates.json`):
```json
{
  "generated": "2026-06-26",
  "us":    [ {"city":"Washington DC","cc":"us","cityCode":"was","ip":"185.213.193.127","seedMs":25} , ... ],
  "nonus": [ {"city":"Montreal","cc":"ca","cityCode":"mtr","ip":"146.70.198.66","seedMs":37} , ... ]
}
```
Full pools (ranked by tonight's measurement):
- **US:** Washington DC (25), Secaucus NJ (28), Boston MA (35), Chicago IL (36),
  Detroit MI (39), Seattle WA (73).
- **Non-US:** Montreal (37), Toronto (43), Querétaro (51), Vancouver (77),
  Bogotá (89), Calgary (89), Lima (108), Tirana (127), Belgrade (128),
  Santiago (132), Kyiv (141), Tel Aviv (159), Buenos Aires (164), Bangkok (252),
  Manila (263).

The pool carries all cities (not just the top 3) so re-ranking can promote a
different city if latencies shift.

## Latency Probe (Approach A: opportunistic + cached)

Accurate latency requires the tunnel down, so the probe is **gated**:

- A timer fires ~every 15 min. The probe runs **only if `mullvad.state == .off`**.
- Additionally, when the existing 5 s `poll()` observes a transition from a
  connected state to `.off`, it triggers an immediate opportunistic probe.
- A probe pings one representative relay per candidate city (concurrency-capped),
  parses min RTT, and — only on success while still `.off` — commits the values
  with a timestamp to the `LatencyStore`.
- When connected, the probe is skipped and the cached ranking is shown. (Refinement,
  optional in v1: the *currently connected* city's relay IP bypasses the tunnel and
  could be refreshed even while connected — noted, not required.)
- The store is persisted to Application Support and seeded from `candidates.json` on
  first run, so the menu is correct immediately and across restarts.

A small disabled footer row shows freshness, e.g. `measured 2h ago (direct)`.

## Architecture

Logic lives in `VPNDNSCore` (pure, unit-tested); AppKit/menu wiring stays in
`Sources/VPNDNSMenuBar/main.swift`.

- **`RelayCandidates.swift`** (core)
  - `struct CandidateRelay { city, cc, cityCode, ip, seedMs }`
  - `struct CandidatePool { us: [CandidateRelay]; nonus: [CandidateRelay] }`
  - `loadCandidates(from:) -> CandidatePool` (decode bundled JSON)

- **`Latency.swift`** (core)
  - `parsePingMinRTT(_ output: String) -> Double?` — pure parser of `/sbin/ping`
    output (handles success, packet loss, missing round-trip line, garbage).
  - `struct CityLatency { city, cc, cityCode, ms, measuredAt }`
  - `final class LatencyStore` — city→latest measurement; `topCities(region:n:) ->
    [CityLatency]` (sorted asc, unreachable last, falls back to seed); Codable
    persistence to Application Support; seeded from `CandidatePool`.

- **`FastCities.swift`** (core)
  - `currentCityCode(fromRelay relay: String?) -> String?` — parse `xx-CITY-...`.
  - `enum ToggleAction { case connect(cc:String, cityCode:String); case disconnect }`
  - `toggleAction(currentRelay:clickedCityCode:clickedCC:) -> ToggleAction`.

- **`main.swift`** (app)
  - `LatencyProbe`: owns the 15-min timer, gating, ping fan-out (via `Shell.run`
    on a background queue, concurrency cap ~8), writes to `LatencyStore`, requests
    icon/menu refresh.
  - `build(_:)`: renders the two sections (header rows disabled; 3 city items each
    with title `"<City> — <ms> ms"` and a checkmark when it's the current city) and
    the freshness footer.
  - Toggle handler: maps the clicked item → `toggleAction(...)`, runs
    `mullvad disconnect` or `mullvad relay set location <cc> <cityCode>` +
    `mullvad connect`, then re-polls so the dot + checkmark update.

- **`scripts/refresh-candidates.sh`** — regenerates `Resources/candidates.json`.

## Error Handling

- ping failure/timeout → city treated as unreachable for that cycle; ranking falls
  back to last good value, else seed. If an entire probe yields nothing, the cache
  is left untouched.
- `mullvad`/`ping` binary missing → probe is a no-op; seeded/cached values shown.
- Corrupt/missing persisted store → fall back to JSON seeds.
- Toggle command failure → next `poll()` reflects true state (no optimistic UI lie).

## Alternatives Considered

- **Probe through the tunnel (B):** rejected — ranking is wrong whenever connected.
- **Split-tunnel-excluded probe (C):** truly live but fragile setup; deferred.
- **Live No-ID list parsing (P-B):** auto-syncs with list edits but heavy/fragile
  parsing for v1; deferred in favor of the regen script.
